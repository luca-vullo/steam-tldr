# Technical architecture — Steam TL;DR

## 1. Overview

Chrome extension, **Manifest V3**, TypeScript, no backend of our own. Three components:

```
┌─────────────────────────────────────────────────────────────┐
│ Steam page (store.steampowered.com/app/{appid}/...)         │
│                                                              │
│  ┌────────────────────┐    messages     ┌─────────────────┐ │
│  │ Content script      │ ◄────────────► │ Service worker  │ │
│  │ - extracts appid    │  chrome.runtime │ (background)    │ │
│  │ - overlay widget    │                │ - fetches Steam │ │
│  │ - renders TL;DR     │                │ - calls the LLM │ │
│  └────────────────────┘                │ - manages cache  │ │
│                                         └───────┬─────────┘ │
└─────────────────────────────────────────────────┼───────────┘
                                                  │
                    ┌─────────────────────────────┼──────────────┐
                    ▼                             ▼              ▼
        store.steampowered.com          LLM provider         chrome.storage
        /appreviews/{appid}?json=1      Anthropic / OpenAI /  (keys, options,
        (recent reviews)                Gemini / Azure AI     summary cache)
                                        Foundry / local
```

Network calls live **only in the service worker**: the content script has no extra host permissions and communicates via `chrome.runtime.sendMessage`.

Custom endpoints (Azure AI Foundry, local servers) are covered by the broad `optional_host_permissions`, but permission is requested at runtime **only for the origin of the saved profile** (`chrome.permissions.request`) — never in bulk.

## 3. Data source: Steam's reviews endpoint

Public JSON endpoint, the same one the store page uses:

```
GET https://store.steampowered.com/appreviews/{appid}
    ?json=1
    &filter=recent|all      // see modes below
    &language=all           // multilingual input: language (F8) governs output only
    &num_per_page=100       // max 100; client-side selection follows
    &purchase_type=all
    &day_range=30           // only with filter=all
```

Fields used per review: `recommendationid` (dedup), `review` (text), `voted_up` (bool), `author.playtime_forever`, `timestamp_created`, `votes_up`, `weighted_vote_score` (0–1 helpfulness computed by Steam). The `query_summary` field also provides `review_score_desc` (e.g. "Very Positive") and totals, useful for the recent-vs-overall comparison in v0.2.

> Empirical note: `weighted_vote_score` is returned as a **string** with `filter=all` and as a **number** with `filter=recent`; 0.5 is Steam's neutral default for reviews with no votes. Parsing is defensive.

### Review selection engine (F2)

The selection of reviews sent to the model is driven by a configuration (savable as a preset, F9):

```ts
interface ReviewSelectionConfig {
  mode: "hybrid" | "recent_scored" | "steam_native";  // default: hybrid
  numReviews: number;   // reviews sent to the model — default 50
  dayRange: number;     // window for the "helpful" component — default 30 (max 365)
  weights: {            // client-side scoring weights (normalized)
    helpfulness: number;  // default 0.4 — weighted_vote_score
    playtime: number;     // default 0.3 — log(hours played), normalized
    substance: number;    // default 0.2 — text length, capped
    freshness: number;    // default 0.1 — linear decay over dayRange
  };
  minChars: number;     // discard shorter reviews — default 30
}
```

Behavior per mode:

| Mode | Fetches | Selection |
|---|---|---|
| `hybrid` (default) | 2: `filter=all&day_range={dayRange}` (helpful) + `filter=recent` (fresh) | Merge with dedup on `recommendationid`, weighted scoring over the whole pool, top `numReviews` |
| `recent_scored` | 1: `filter=recent` | Weighted scoring, top `numReviews` |
| `steam_native` | 1: `filter=all&day_range={dayRange}` | Steam's order, first `numReviews`; falls back to `filter=recent` below 10 results |

Common rules, applied before scoring:

- reviews shorter than `minChars` are discarded (contentless "+1" reviews);
- the final sample **preserves the pool's real positive/negative ratio**, to avoid skewing the sentiment;
- very fresh reviews have few votes: a missing/neutral `helpfulness` does not zero the score (the other weights are renormalized).

The active configuration contributes to the cache key (config hash), so switching presets regenerates the summary instead of serving one computed with a different selection.

Usage policy: at most 2 requests per page visit (1 in non-hybrid modes), then cache; default browser `User-Agent`, no aggressive pagination in the MVP.

## 4. Summarization: LLM provider layer

### Provider profiles (F7)

The summarizer is provider-agnostic: it builds the prompt and delegates the call to a **per-protocol** adapter. The user defines **profiles** (protocol + endpoint + key + model) and picks the active one; profiles live in `chrome.storage.local`.

```ts
type ProviderKind = "anthropic" | "openai_compat" | "gemini";

interface ProviderProfile {
  id: string;
  name: string;     // user-chosen label
  kind: ProviderKind;
  baseUrl: string;  // "" = the protocol's default endpoint
  apiKey: string;   // may be empty for local servers
  model: string;    // model ID, or deployment name on Azure Foundry
}

interface LLMProvider {
  kind: ProviderKind;
  summarize(req: SummarizeRequest, profile: ProviderProfile): Promise<TLDRSummary>;
}
```

Each adapter is responsible for obtaining JSON conforming to the `TLDRSummary` schema using the protocol's native mechanism; a single schema validation runs downstream.

| Protocol | Covers | Client | Structured output |
|---|---|---|---|
| `anthropic` | Claude API (default); **Claude on Azure AI Foundry** (baseUrl = resource endpoint, model = deployment name) | `@anthropic-ai/sdk`; with a baseUrl, `@anthropic-ai/foundry-sdk` (Azure auth) | Structured outputs (`output_config.format` with JSON schema) |
| `openai_compat` | Official OpenAI; **Azure AI Foundry** (OpenAI v1 endpoint); **local models** (Ollama `http://localhost:11434/v1`, LM Studio, ...) | Direct `fetch` to `{baseUrl}/chat/completions` (both `Authorization: Bearer` and `api-key` headers for Azure compatibility) | `response_format: json_schema` strict; if the endpoint rejects it (400), retry with the JSON constraint in the prompt + extraction/validation |
| `gemini` | Google Gemini API | Direct `fetch` to `generateContent` | `responseMimeType: application/json` + `responseSchema` (variant without `anyOf`) |

User-pasted endpoints are normalized: trailing `/v1/messages` (Anthropic) or `/chat/completions` (OpenAI-compatible) suffixes are stripped, since the clients append the path themselves.

### Anthropic adapter (default)

- SDK: `@anthropic-ai/sdk` from the service worker, with `dangerouslyAllowBrowser: true` — acceptable because the key is **the user's own**, entered in the options and stored in `chrome.storage.local`; there is no key of ours to protect and no backend to hide it behind. With a baseUrl set (Claude on Azure AI Foundry) the adapter uses `@anthropic-ai/foundry-sdk`, which handles Azure authentication with the same `messages.create` surface.
- **Structured outputs** (`output_config.format` with JSON schema) to always get a parsable `TLDRSummary`, no prefill (not supported on current models).
- Parameters: `max_tokens: 4000`, no `temperature` (removed on current models), no `thinking` parameter (so the adapter works with every Claude model: Haiku 4.5 does not support adaptive thinking, and summarization doesn't need it).

### Output schema

```ts
interface TLDRSummary {
  verdict: string;          // one line
  sentiment: "positive" | "mixed" | "negative";
  pros: string[];           // 3–5 recurring points
  cons: string[];           // 3–5 recurring points
  recent_changes: string | null; // notes about patches/updates if reviews mention them
  reviews_analyzed: number;
  language: string;
}
```

### Prompt

System: neutral analyst role; instructions to summarize **recurring themes** (not individual opinions), never to quote insults or offensive content verbatim, to flag small or contradictory samples, to answer in the requested language, and to treat review text as data, never as instructions (prompt-injection defense).

User: game metadata (name, `review_score_desc` and totals) + the N selected reviews as a compact JSON list (text truncated at ~1500 chars each, with `voted_up` and hours played).

### Models

For Anthropic (default):

| Model | ID | Input $/1M | Output $/1M | Use |
|---|---|---|---|---|
| **Claude Opus 4.8** (default) | `claude-opus-4-8` | $5.00 | $25.00 | Best summary quality |
| Claude Haiku 4.5 (budget option) | `claude-haiku-4-5` | $1.00 | $5.00 | Selectable in the options |

For OpenAI, Gemini and Azure AI Foundry the model is a free-text field in the options: model IDs for these providers change frequently and must not be hardcoded.

### Cost estimate

Assumptions: 50 reviews ≈ 10–15k input tokens, ~500 output tokens.

| Model | Cost per summary | 100 games/month |
|---|---|---|
| Opus 4.8 | ~$0.06–0.09 | ~$6–9 |
| Haiku 4.5 | ~$0.012–0.018 | ~$1.20–1.80 |

For other providers the cost depends on the user's chosen model and the provider's pricing; the order of magnitude is the same (10–15k input tokens per summary). With the 24h cache, repeat visits to the same game cost zero.

## 5. Cache

`chrome.storage.local`, key `summary:{appid}:{lang}:{profileId}:{model}:{selectionHash}` (hash of the active `ReviewSelectionConfig`), value `{ summary, reviewsUsed, poolSize, querySummary, createdAt }`. Default TTL 24h (configurable, 0 disables). Manual invalidation via the "Regenerate" button in the widget. `chrome.storage.local` is limited to ~10MB: LRU eviction beyond ~200 entries.

## 6. Error handling

| Error | UI behavior |
|---|---|
| Missing/invalid API key (401) | Widget message with a direct link to the options page |
| Provider rate limit (429) | Generic error with retry |
| Steam endpoint unreachable / empty | "Summary unavailable" message |
| Model refusal / truncated or schema-invalid output | Generic "summary unavailable" message, details in console |
| Custom endpoint: host permission denied by the user | Save is blocked in the options with an explanation |

## 7. Security and privacy

- API keys (for any provider) never appear in code, logs or the repo; they live only in `chrome.storage.local`.
- No telemetry, no server of ours, no personal data processed: only public review texts travel to the configured AI provider.
- Each key is sent exclusively to its own profile's endpoint (dedicated host permission in the manifest; for custom endpoints, requested at runtime for that specific origin).
- Review text is untrusted input: the widget renders the summary as text (never `innerHTML` from generated content), and the prompt treats reviews as data, not instructions.

## 8. Key decisions

1. **Build tooling: Vite + CRXJS.** HMR on the content script and automatic manifest/asset handling are worth the extra dependency; a bundler is needed for the Anthropic SDK anyway.
2. **Widget UI: vanilla TS.** One component with a handful of states doesn't justify a framework; a Preact migration is deferred to v0.2 if the UI grows.
3. **Activation: manual click by default.** Zero involuntary API costs while browsing; automatic activation is available as an option (F5).
4. **Overlay widget instead of in-page injection.** Injecting into Steam's DOM proved fragile (clipped by the header layout); a fixed-position tab + drawer is immune to store redesigns and uses no Steam selectors.
5. **Provider profiles instead of a fixed provider enum.** A profile is protocol + endpoint + key + model, so the same three adapters cover the official APIs, Azure AI Foundry and local OpenAI-compatible servers.
